# Системный промпт: сопутствующая документация (`app/docs/`)

## Контекст

- Код и точка входа — в **`app/`** (не в `app/docs/`).
- `app/docs/` — каноническое хранилище **сопутствующей** документации по приложению.
- При изменениях логики, конфигурации, зависимостей или наблюдаемого поведения обновляй документы **только там, где это зона ответственности файла** (см. матрицу ниже). Цель — восстановить проект без полного чтения кода.

## Принципы

1. Документ отражает **актуальное** состояние; устаревший текст хуже отсутствующего — сверяйся с кодом в `app/` и правь.
2. В доках — **контракты**: вход/выход, CLI, конфиг, ошибки, интеграции. Не пересказывать реализацию построчно.
3. Секреты не коммитить; в примерах — плейсхолдеры и имена переменных окружения.
4. **Один источник истины** на тему: остальные файлы дают ссылку, а не копипасту.

## Матрица: что за что отвечает

| Файл | Ответственность | Обновлять при |
|------|-----------------|---------------|
| `app/docs/README.md` | Оглавление документации, навигация по `docs/`, ссылки на запуск и типовые проблемы | Появлении/переименовании документов в `docs/` |
| `app/README.md` или `app/README.MD` | Минимальный быстрый старт рядом с кодом (если файл есть) | Смене команд запуска/установки; в конце — ссылка на `app/docs/README.md`, без дублирования длинных инструкций |
| `app/docs/CHANGELOG.md` | Версия (semver или дата), изменения, известные ограничения | Заметном изменении поведения, CLI, логов, конфига, контрактов API |
| `app/docs/ARCHITECTURE.md` | Потоки данных, модули, интеграции, инварианты, расширение | Новом слое/провайдере, смене схемы данных, нетривиальных очередях/ретраях |
| `app/.env.example` | Список переменных и смысл (без значений-секретов) | Новой/переименованной переменной, смене обязательности или формата |
| `app/docs/CONTRIBUTING.md` | Процесс правок, стиль, проверки перед merge | Смене toolchain, линтеров, обязательных тестов/pre-commit |
| `app/docs/ROADMAP.md` | Планы вперёд (не история) | Смене приоритетов, закрытии эпиков |
| `app/docs/troubleshooting.md` | Симптом → причина → шаги (проверенные сценарии) | Повторяющихся или неочевидных сбоях с известным решением; после смены поведения — пометить/убрать устаревшее |

**CHANGELOG vs ROADMAP:** в `CHANGELOG` — что уже сделано/выпущено; в `ROADMAP` — что планируется.

## После правок в коде `app/`

- Затрагивает **пользователя** (запуск, конфиг, вывод, CLI) → `app/.env.example`, запись в `app/docs/CHANGELOG.md`, при необходимости строка в `app/docs/README.md` или кратко в `app/README*`.
- Затрагивает **устройство системы** → `app/docs/ARCHITECTURE.md`.
- Повторяющийся сбой с решением → `app/docs/troubleshooting.md`.
- Чистый внутренний рефакторинг без смены поведения → доки не трогать, если тексты остаются правдивыми.

## `app/docs/troubleshooting.md`

- Формат записи: **симптом** (цитата ошибки или поведение) → **причина** → **шаги** → **как проверить**; при необходимости ссылка на файлы в `app/`.
- Императив, короткие шаги; фрагменты логов — только узнаваемые, не целый `debug.log`.
- Один симптом — одна карточка; несколько причин — ветвление «если … то …».
- Устаревшее: пометка `Устарело с <версия или дата>` и ссылка на актуальный раздел; удалять, если поведение вырезано из кода.
- Не дублировать длинные туториалы — отсылка к `app/docs/README.md` / `app/README*`.

## Запреты

- Не создавать новые `.md` в репозитории без явного запроса пользователя, кроме случая, когда иначе задачу закрыть нельзя — тогда один файл с ясным назначением.
- Не размножать один и тот же текст в трёх местах: один канонический абзац, остальное — ссылки.
