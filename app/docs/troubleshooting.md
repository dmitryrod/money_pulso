# Troubleshooting

Формат: **симптом** → **причина** → **шаги** → **проверка**.

## При старте контейнера: ошибки подключения к PostgreSQL

**Симптом:** в логах `app` — `connection refused`, `could not connect to server`, таймаут к хосту `postgres`.

**Причина:** приложение стартовало до готовности БД или неверные `POSTGRES_*` / сеть Docker.

**Шаги:**

1. Убедиться, что `postgres` в `docker compose ps` в состоянии healthy.
2. Проверить в `.env` согласованность `POSTGRES_USER`, `POSTGRES_PASSWORD`, `POSTGRES_DB` с тем, что ожидает сервис `postgres` и приложение.
3. Перезапустить стек: `docker compose down` и `docker compose up`.

**Проверка:** в логах после старта нет повторяющихся traceback на `Database init failed`; админка открывается.

## Обзор (`/admin/`) долго без цифр или высокий TTFB

**Симптом:** после обновления страницы «Обзор» пустые/прочерки или документ HTML приходит через многие секунды.

**Причина (типично):** раньше — последовательные тяжёлые COUNT по большой таблице `signals` без pg_stat и полный `os.walk` каталога для метрики размера; медленная сеть до Postgres.

**Шаги:** использовать актуальный код (`app/admin/dashboard_summary.py`, `monitoring_metrics.record_snapshot_for_dashboard`, клиентский ключ **`sessionStorage` `mp_admin_dashboard_summary_v1`**); полный скан размера каталога `app/` остаётся на странице **«Система»** / частом polling **`/admin_api/monitoring/metrics`**.

**Проверка:** во вкладке Network время загрузки документа HTML заметно снижается; при повторных F5 в пределах ~5 с возможен ответ из TTL-кэша процесса; после первого визита — данные из `sessionStorage` до ответа сервера.

## После входа в админку: модалка Chrome «Смените пароль» / пароль в утечке

**Симптом:** сразу после отправки формы `/admin/login` (часто при демо-логине **DEMO** / **DEMO**) браузер Google Chrome показывает окно вроде «Пароль … был раскрыт в результате утечки данных» / «Смените пароль».

**Причина:** это **не код приложения** (не Jinja, не `starlette-admin`). Встроенная проверка **Google Chrome** / менеджера паролей: введённый пароль совпал с базой известных скомпрометированных паролей. Типовые короткие пары попадают в эту проверку чаще.

**Шаги:**

1. **На стороне сервера:** задать в env для демо уникальный сильный **`DEMO_PASSWORD`** (и при необходимости **`DEMO_LOGIN`**), перезапустить приложение — предупреждение обычно пропадает.
2. **На стороне браузера:** настройки Chrome → автозаполнение и пароли → при необходимости отключить предупреждения о скомпрометированных паролях или не сохранять пароль для этого сайта (формулировки зависят от версии Chrome).
3. Не искать это сообщение в `app/admin/templates/` или в API — его **нельзя отключить из кода** Money Pulso.

**Проверка:** после смены `DEMO_PASSWORD` на длинную случайную строку повторный вход без модалки (или только подтверждение сохранения пароля, без предупреждения об утечке).

## Падение при старте: `ImportError: cannot import name 'SignalCounter'`

**Симптом:** контейнер `app-1` постоянно перезапускается (`exited with code 1`). В логах терминала ошибка: `ImportError: cannot import name 'SignalCounter' from partially initialized module 'app.utils' (most likely due to a circular import)`. Часто перед этим видно сообщение `Failed to fetch CoinMarketCap listings: ... SSLError`.

**Причина:** циклический импорт между внутренними модулями (`app.utils`, `app.screener`, `app.admin`). Ошибка сети от CoinMarketCap выступала лишь триггером, который приводил к загрузке модулей в неудачном порядке, из-за чего инициализация прерывалась.

**Шаги:**

1. Проблема исправлена в коде (убран циклический импорт через `TYPE_CHECKING` в `format_filter_failure.py`).
2. Убедитесь, что вы используете актуальную версию приложения.
3. Сама по себе ошибка от CoinMarketCap не является фатальной — при правильной структуре импортов она просто запишется в лог (как предупреждение), и приложение продолжит работу.

**Проверка:** после обновления кода приложение успешно стартует и продолжает работу, даже если первый запрос к CoinMarketCap завершился таймаутом или ошибкой SSL.

## `ValueError: Invalid environment`

**Симптом:** падение при импорте/старте, сообщение про неверный `ENVIRONMENT`.

**Причина:** в `ENVIRONMENT` указано не `development` и не `production`.

**Шаги:** задать `ENVIRONMENT=development` или `ENVIRONMENT=production` в `.env` или окружении процесса.

**Проверка:** приложение поднимается, в логе startup видно ожидаемое окружение.

## Долгая работа: pump / liquidations «замолчали», OI ещё шлёт

**Симптом:** после многих часов без перезапуска контейнера сигналы только от OI.

**Причина (частые):** OI идёт через REST; aggTrades и liquidations — через WebSocket. Редкие «залипания» потока не всегда попадают под watchdog по возрасту `last_update_ts`.

**Шаги:**

1. Recycle по умолчанию **включён** (**14400** с), если `WS_PERIODIC_RECYCLE_SEC` не задана или пустая (и в Docker, и при локальном `uvicorn`). В логах при старте: `WebSocket periodic recycle: every 14400s`. Полностью выключить: `WS_PERIODIC_RECYCLE_SEC=0` в `.env`. Первый интервал отсчитывается от момента появления пары парсеров (сразу после старта рестарта нет).
2. Проверить пороги ликвидаций / pump (см. карточку ниже «Только OI…»).

**Проверка:** в логах периодически появляется `Periodic WS recycle`; после всплеска волатильности снова приходят сигналы pump/liquidations.

## В `app.log` спам: `liquidations sum filter` и `amount_usdt: 0.0`

**Симптом:** десятки/сотни строк вида `[x] BTCUSDT. Reason: liquidations sum filter. Details: ... 'amount_usdt': 0.0 ...` за один проход цикла.

**Причина:** фильтр отработал: за `lq_interval_sec` по этому символу сумма ликвидаций в \$ ниже порогов (`lq_min_amount_usd` / `%` от суточного объёма) или в окне не было событий. При **живом** WebSocket это нормально для тихого рынка и жёстких порогов (например 50 000 \$ за 180 с). Не путать с «отвалился поток» — тогда в debug вырос бы `parsers_age_sec.liquidations` (десятки секунд и выше без обновления).

**Шаги:** временно снизить `lq_min_amount_usd` / `lq_min_amount_pct` или включить debug и сверить возраст парсеров и `liquidations_symbols`.

**Проверка:** при крупных ликвидациях на рынке появляются сигналы; в debug возраст `liquidations` остаётся небольшим.

**Примечание:** при включённом debug у скринера построчные отказы фильтров в `app.log` — уровень **DEBUG**; сводка — `cycle_end` в `debug.log`. Чтобы снова писать их в `app.log`, задай в `.env` **`LOG_LEVEL=DEBUG`** (см. `app/config/logger.py`). При старте подхватывается `app/.env` только для переменных, которых **ещё нет** в окружении процесса: если `LOG_LEVEL` уже задан в системе / в `docker-compose` / в shell, значение из файла не перезапишет его.

## Только OI шлёт сигналы, pump / liquidations — нет (долгое время)

**Симптом:** по Telegram идут сигналы только от скринера с OI; ликвидации и pump «молчат» часами.

**Причина (частые):** OI обновляется REST-пуллингом (`OpenInterestParser`), pump/volume завязаны на свечи из aggTrades WebSocket, ликвидации — на отдельный WebSocket. Тихий рынок + жёсткие пороги дают нули в фильтре ликвидаций (`amount_usdt: 0` в debug). Раньше возможен был рассинхрон ключа тикера в кеше ликвидаций и в `Consumer` — в актуальной версии ключи выровнены через `normalize_ticker` (`liquidations.py`).

**Шаги:**

1. При включённом debug у скринера смотреть `cycle_end` / `cycle_start`: `parsers_age_sec.liquidations`, `parsers_age_sec.agg_trades` — если > ~90 с, сработает watchdog перезапуска WS (см. `operator.py`).
2. Для ликвидаций проверить пороги (`lq_min_amount_usd` / `%`); при нулевой активности сигналов не будет даже при живом потоке.
3. В логах искать `Watchdog: stale websocket` / `restarted websocket parser`.

**Проверка:** после всплеска ликвидаций на рынке или ослабления порогов снова появляются сообщения; возраст парсеров в debug остаётся небольшим.

## Нет сигналов в Telegram при «включённом» скринере

**Симптом:** скринер enabled, но сообщений нет.

**Причина (частые):** не заполнен ни один фильтр полностью; несовместимость рынка и фильтра (фьючерсные фильтры на споте); слишком жёсткие пороги; **не задана пара** `chat_id` **и** `bot_token` (ни в скринере, ни в `.env`) — тогда Telegram намеренно не используется; пустой/ошибочный токен или чат при попытке отправки; лимиты «макс. сигналов в сутки» / таймаут между сигналами.

**Шаги:**

1. В админке проверить, что у хотя бы одного фильтра заполнены **все** обязательные поля.
2. Сверить тип рынка с типом фильтров (OI, funding, liquidations — фьючерсы).
3. Проверить токен бота и ID чата (в т.ч. через @BotFather / бот для получения id).
4. Временно ослабить пороги и увеличить лимиты сигналов для проверки канала.

**Проверка:** при включённой отладке у скринера в сообщениях/логах появляется больше диагностики; тестовое сообщение боту в чат проходит.

## В логе есть сигнал, в Telegram пусто

**Симптом:** в `debug.log` событие `signal` с текстом сообщения или в `app.log` строка `Signal created`, а в чате ничего не пришло.

**Причина 1:** событие **`signal` в debug** пишется **до** отправки в Telegram (параллельная задача). Смотри **`telegram_sent`** / **`telegram_error`** или `signals_log.txt` с `telegram_ok` и полем `error`.

**Причина 2:** ответ Bot API с **`ok: false`** при HTTP 200 (до исправления не считался ошибкой): неверный **`chat_id`**, пользователь не нажал **Start** у бота, бот удалён из чата, **невалидный HTML** в шаблоне (`parse_mode=HTML`), слишком длинный текст и т.д. Текст ошибки — в `description` ответа API (теперь в исключении / `signals_log.error`).

**Причина 3:** скринер без настроенного Telegram: в `signals_log` / БД сигнал с **`telegram_ok: false`** и без поля `error` (отправка пропущена). В **`debug.log`** при отладке скринера — событие **`telegram_skipped`**.

**Шаги:** сверить `chat_id` и токен; отправить тест через API или @BotFather; упростить шаблон/выключить HTML; смотреть `telegram_error`, `telegram_skipped` и `signals_log.txt`.

**Проверка:** в `signals_log.txt` у успешной доставки `telegram_ok: true` и есть `message_id` в `telegram`.

## `Logging error` / `FileNotFoundError` при ротации `debug.log` / архивов

**Симптом:** в stderr при записи JSON-debug — traceback на `doRollover`, `rename` для `app/logs/debug.…log` (раньше встречались имена вида `debug.log.1`).

**Причина:** гонка или том Docker (архив удалён снаружи, несколько инстансов, очистка логов).

**Шаги:** актуальная версия — `_SafeMoscowSizeRotatingFileHandler` в `config/debug_json_logger.py`; не удалять вручную `debug.*.log` при работающем процессе.

**Проверка:** ошибка ротации не повторяется; при сбое rollover в логе одно предупреждение и запись продолжается.

**Симптом:** логи `wait_for_internet`, редкие циклы Operator.

**Причина:** штатное поведение `app/utils/connectivity.py` для временных сетевых ошибок.

**Проверка:** после восстановления сети циклы Operator снова регулярные, парсеры обновляют данные.

## Админка: очень долгая загрузка / NO_FCP в Lighthouse, в Network «inter.css» (failed) десятки секунд

**Симптом:** страницы `/admin/...` визуально пустые долго; в Network запрос к **`inter.css`** или **`rsms.me`** со статусом `(failed) net::ERR_CONNECTION...`, время **10–40+ с**; Lighthouse **NO_FCP**.

**Причина:** в шаблоне **starlette-admin** (`base.html`) в `<head>` был **`@import url('https://rsms.me/inter/inter.css')`**. Ресурс блокирует отрисовку; при недоступности хоста (фаервол, DNS, корпоративная сеть) браузер ждёт таймаут TCP. Фильтр `rsms.me` только в `_additional_css_links` у `SettingsModelView` **не убирает** этот `@import` (он в другом месте шаблона).

**Шаги:** в проекте переопределён **`app/admin/templates/base.html`** без внешнего `@import`, шрифт — системный стек (см. комментарий в файле). Убедиться, что в Network нет запросов к `rsms.me`.

**Проверка:** document остаётся быстрым; первая отрисовка без многосекундной паузы; Lighthouse даёт FCP/LCP.

## Lighthouse SEO: «Page is blocked from indexing» / низкий балл SEO

**Симптом:** в категории SEO красным отмечено, что страница закрыта от индексации; балл падает (например ~60).

**Причина:** в **production** в `layout.html` выводится `<meta name="robots" content="noindex, nofollow">` — это **намеренно** для админки. Lighthouse трактует это как провал SEO-аудита, хотя для внутреннего UI это норма.

**Поведение:** при **`ENVIRONMENT=development`** мета `robots` **не** добавляется (удобно для прогона Lighthouse на localhost). При **`ENVIRONMENT=production`** — `noindex` остаётся. Глобаль шаблона: `admin_robots_noindex` в `register_admin_routes`.

**Проверка:** на dev нет тега `robots` в исходнике страницы; на prod он есть.

## Админка: долгие ответы при переходах между страницами

**Симптом:** заметная пауза (порядка секунд и выше) при открытии раздела **«Система»** (`/admin/monitoring`) или «всё подвисает» на время запроса к этой странице при одном воркере uvicorn.

**Причина (типичная):** в `MetrCustomView` раньше вызывался `psutil.cpu_percent(interval=1)` **синхронно** внутри `async def render`, что блокировало event loop на ~1 с. Страница **«Логи»** читает **весь** `app/logs/app.log` в память — при большом файле ответ может быть очень тяжёлым.

**Шаги:**

1. Убедиться в DevTools → Network: большой **Waiting (TTFB)** у HTML, а не только загрузка статики.
2. Проверить размер `app/logs/app.log`; при необходимости ротировать/очищать лог.
3. Использовать актуальный код: CPU для «Система» считается в thread pool с коротким интервалом (см. `app/admin/view.py`).

**Проверка:** переход в «Система» не блокирует параллельные запросы на секунды; TTFB страницы мониторинга сопоставим с ~0.1–0.3 с плюс сеть, а не ~1 с только из-за CPU.

**Дополнительно (рост данных / логов):** если снова «подвисает» раздел **«Логи»** — проверьте размер `app/logs/app.log`; в актуальной версии в UI показывается только **хвост файла (~512 KiB)**. Если тормозит страница **«Сигналы»** при большой таблице `signals` — для подсчёта страниц используется быстрая оценка из `pg_stat_user_tables` (возможна погрешность относительно точного `COUNT(*)` до ближайшего `ANALYZE`).
