# Abbreviation Map

## Policy

- Prefer canonical abbreviations for repeated domain terms.
- Do not abbreviate if it creates ambiguity for first-time readers.
- Keep literals unchanged: code, commands, URLs, file names, env vars.
- Expand on first use when abbreviation is uncommon.

## Canonical Abbreviations

- **Markdown** -> `MD`
- **reference** -> `ref`
- **references** -> `refs`
- **configuration** -> `config`
- **application** -> `app`
- **environment** -> `env`
- **requirements** -> `reqs`
- **requirement** -> `req`
- **constraints** -> `constraints` (no abbreviation; preserve clarity)
- **information** -> `info`
- **example** -> `ex`
- **examples** -> `exs`
- **approximately** -> `approx`
- **minimum** -> `min`
- **maximum** -> `max`
- **temporary** -> `temp`
- **identifier** -> `id`
- **identifiers** -> `ids`
- **number** -> `num`
- **numbers** -> `nums`
- **parameter** -> `param`
- **parameters** -> `params`
- **function** -> `fn`
- **functions** -> `fns`
- **message** -> `msg`
- **messages** -> `msgs`

## Allowed Shortforms for Common Phrases

- **for example** -> `e.g.`
- **that is** -> `i.e.`
- **versus** -> `vs`
- **with respect to** -> `re`
- **and so on** -> `etc.`

## Avoid

- Inventing new abbreviations not in this map unless locally defined.
- Stacking too many abbreviations in one sentence.
- Abbreviating headings into unreadable fragments.
- Replacing words with symbols outside [symbol-and-punctuation-policy.md](symbol-and-punctuation-policy.md).
