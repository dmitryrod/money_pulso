# Stat: столбцы low/high за минуту на той же оси времени

## Универсальный источник цены (не PD)

**`pd.current.final_price` не подходит как единственный источник:** фильтр PD может быть выключен — тогда в сэмплах нет цены, графики и столбцы пустые.

**Ограничение (зафиксировано): отдельный WebSocket под цены — делать нельзя.** Цена берётся только из уже существующего пайплайна (тот же цикл, что и фильтры).

В [`Consumer._check_filters`](app/screener/consumer.py) данные приходят из `agg_trades.fetch_collected_data()` → `klines`; в [`evaluate_test_mode_snapshot`](app/screener/test_mode_eval.py) уже считается `last_price = float(klines[-1]["c"])`. Цена **уже** есть в момент оценки, она просто **не сериализуется** в JSONL.

**Плотность выше частоты сэмплов ([`maybe_persist_sample`](app/screener/scanner_runtime.py), ~5 с) не требуется** — отдельные тики, ускоренная запись цены и т.п. **вне scope**.

**Минимально универсально:** добавить в тестовый payload и в [`build_sample_line`](app/screener/scanner_runtime.py) поля цены из того же цикла, что и `score` / `test_filters` (например `last_price`, опционально `price_ts`). Источник один для всех конфигов скринера, без зависимости от PD.

---

## Формат записи цены (единственный в scope)

**Только подход A:** в каждой строке `kind:sample` хранить `last_price` (+ опц. `price_ts`), из того же цикла, что и остальная статистика. Частота = частота сэмплов (~5 с), **без** отдельных `price_tick`, **без** доп. OHLC-строк в JSONL.

Для графиков с большим таймфреймом на фронте: агрегировать уже сохранённые точки (rollup по минутам/5m из массива сэмплов) — схема полей не меняется.

Варианты B/C из прежних черновиков **не планируются** (не нужна большая плотность; отдельный WS запрещён).

---

## Решение по UX

- Один **mixed** Chart.js: линии Score / PD / OI / LQ как сейчас + **dataset типа `bar`** (floating bars) для диапазона цены за минуту.
- Ось **yPrice** (цена в USDT), отдельно от `y` (score) и `y1` (%).
- Порядок отрисовки: `order` у dataset — столбцы «под» линиями.

## Хватает ли сейчас данных?

| Что нужно | Есть ли в JSONL |
|-----------|-----------------|
| Метка времени минуты | Да (`ts` у каждого `sample`) |
| Несколько наблюдений цены внутри минуты | Частично: heartbeat ~5 с ([`maybe_persist_sample`](app/screener/scanner_runtime.py)), в минуте обычно **несколько** точек |
| Цена в точке | После **фазы 1** (план выше): `last_price` в каждой строке `sample` из того же цикла, что и фильтры — **не зависит от PD**. До внедрения — только костыль через PD `final_price` |

**Истинные** биржевые high/low за минуту **не хранятся**: нет ни минутных свечей, ни полного тикового ряда в статистике. Можно строить только:

- **Диапазон по сэмплам скринера за минуту:**  
  `price_low = min(p_i)`, `price_high = max(p_i)` по всем сэмплам в той же UTC-минуте, где `p_i` = **`sample.last_price`** (универсально после фазы 1).

Это **не** гарантированно совпадает с OHLC биржи; при одном сэмпле на минуту столбик вырождается в линию (low = high). В режиме «все тики» точек больше → диапазон ближе к реальности.

## Алгоритм (фронт достаточно)

1. Разбить сырые `samples` по минутам `mk = floor(Date.parse(ts)/60000)`.
2. Для **линий** (текущее поведение «1 точка/мин»): по каждой минуте — **последний** sample (как сейчас `aggregateSamplesByMinute`).
3. Для **столбцов**: по каждой минуте — среди **всех** сэмплов в минуте взять min/max цены (один проход по отсортированным сэмплам, две карты или одна структура `{ last, minP, maxP }` на минуту).
4. Выравнивание по оси X: те же `labels`, что и у линий (после шага 2), для каждой метки подставить `[low, high]` из шага 3; если цены нет — `null` или пропуск.

Режим «все тики» использует те же сырые сэмплы; для минутной агрегации столбцов логика та же, что и выше (min/max внутри минуты по всем тикам в ней).

## Стоимость реализации

- **Низкая–средняя** (порядка **0.5–1.5 дня** при знакомстве с кодом): правки в основном в [`analytics_stat.html`](app/admin/templates/analytics_stat.html) — одна общая агрегация по минутам, mixed chart, ось `yPrice`, floating bars.
- **Обязательная надбавка для универсальности:** `last_price` (+ опц. `price_ts`) в [`evaluate_test_mode_snapshot`](app/screener/test_mode_eval.py) и [`build_sample_line`](app/screener/scanner_runtime.py) — **без привязки к PD**, без новых WS и без отдельных тиков.

## Риски

- Путаница у пользователя: думать что это биржевые свечи — в UI подпись вроде «диапазон цены по сэмплам скринера (≈5 с), не OHLC биржи».
- Если нужны **настоящие** минутные HL — отдельный эндпоинт klines с биржи (дороже по коду и эксплуатации).
