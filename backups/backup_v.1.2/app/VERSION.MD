# paid-screener — снимок v1.2

**Дата фиксации снимка:** 2026-03-26 (каталог `backups/backup_v.1.2/app`).  
**Версия в `pyproject.toml`:** 0.1.0 (внутренний маркер пакета; релизная нумерация ведётся отдельно.)

---

## 1. Назначение

Веб-приложение на **FastAPI** с админ-панелью (**Starlette-Admin**) для настройки нескольких **скринеров** криптовалютных рынков. Для каждой включённой конфигурации в фоне работает **Consumer**: по данным с бирж (через **unicex**) периодически проверяются фильтры; при срабатывании условий отправляется сообщение в **Telegram**.

База данных: **PostgreSQL** (**asyncpg** + **SQLAlchemy 2** async). Логи приложения: **loguru** → `app/logs/app.log` (ротация).

## 2. Текущий функционал

### 2.1 Админка и API

- Список/создание/редактирование скринеров (настройки в таблице `settings`).
- Маршруты FastAPI для админки и служебные эндпоинты.
- **Глобальный режим отладки** на странице списка скринеров (`/admin/screeners/list`): чекбокс «Глобальный режим отладки» массово включает/выключает флаг `debug` у всех записей в БД (отдельный JS + `GET/POST /admin_api/screeners/global-debug`).

### 2.2 Архитектура скринера (Operator + парсеры + консьюмеры)

- **Operator** раз в несколько секунд подтягивает включённые настройки из БД.
- На каждую уникальную пару **(биржа, тип рынка)** создаётся **один набор парсеров** (`ParsersDTO`), общий для всех скринеров с этой парой:
  - **AggTrades** — агрегированные сделки → свечи (WebSocket).
  - **TickerDaily** — суточная статистика по тикерам (REST, периодический опрос).
  - Для **FUTURES** дополнительно: **FundingRate**, **Liquidations** (WebSocket), **OpenInterest** (REST/батчи в зависимости от биржи).
- На каждый скринер (запись настроек) — отдельный **Consumer** с собственными лимитами, таймаутами, фильтрами и Telegram `bot_token` / `chat_id`.

### 2.3 Фильтры и сигналы

- Проверка **только USDT-пар**, **whitelist/blacklist**.
- **Суточный объём и изменение цены** (24h).
- **Pump/dump** по короткому интервалу и %.
- **Множитель объёма** относительно суточного.
- **Открытый интерес** (OI): % и/или USD за интервал.
- **Фандинг** (диапазон).
- **Сумма ликвидаций** за интервал (USD и/или % от суточного объёма).
- **Таймаут** между сигналами по символу, **лимит сигналов в сутки** по символу.
- Текст уведомления формируется через **`generate_text`** (шаблоны, ссылки на биржу/TradingView/CoinGlass и т.д.).
- Рейтинг **CoinMarketCap** подтягивается фоновым потоком (`coinmarketcap_rank.py`), в тексте может отображаться как `N/A` при сбое API.

### 2.4 Режим отладки (per-screener + глобальный)

- Поле **`debug`** в настройках скринера (и глобальное массовое переключение).
- При `debug=True`:
  - Подробные события в **`app/logs/debug.log`** в формате **одна строка = один JSON** (JSONL).
  - Обязательные поля записи: `ts`, `screener_name`, `screener_id`, `level`, `exchange`, `market_type`, `event`, `symbol`, `payload`, `run_id`, `cycle_id`.
  - Запись идёт через **очередь** (`QueueHandler` / `QueueListener`) и **ротация файла** (аналогично идее app.log: большой лимит размера и несколько архивов).
  - События: `cycle_start` / `cycle_end`, `signal`, `telegram_sent` / `telegram_error`, `error`, при включённом debug у кого-то из скринеров пары — в Operator также `stale_data`, `task_exception`, `consumer_task_done` и т.д.
  - Секреты из env в debug-лог не попадают (Telegram-ответы санируются).

### 2.5 Устойчивость и мониторинг

- **Автоперезапуск задач парсеров**, если `asyncio.Task` завершилась (в т.ч. «тихо» после ошибки в `start()`).
- **Watchdog по «устареванию» websocket-данных**: если эффективный возраст обновлений **agg_trades** или **liquidations** превышает порог (~90 с), выполняется **`stop()` → сброс состояния → новый `start()`** с **экспоненциальным backoff + jitter** между принудительными рестартами одной пары биржа/рынок; при «здоровых» данных backoff сбрасывается.
- **Учёт нестабильного интернета** (`utils/connectivity.py`):
  - классификация **временных сетевых/БД** ошибок;
  - **проверка выхода в интернет** по нескольким HTTPS-URL (настраивается env **`CONNECTIVITY_PROBE_URLS`**);
  - **`wait_for_internet`** — ожидание с растущим интервалом;
  - вызовы при сбоях в цикле Operator и при старте приложения (инициализация БД, `start_exchanges_info`).

### 2.6 Прочее

- Docker / docker-compose для развёртывания.
- Локальная копия/патчи **unicex** в дереве проекта (в т.ч. websocket) — учитывать при обновлении зависимости.

## 3. Последние изменения и обновления (в этом снимке относительно более ранних версий)

1. **Глобальный debug** в UI списка скринеров + API.
2. **JSONL debug.log** с ротацией и асинхронной записью в фоне.
3. **Диагностика «зависаний»**: `stale_data`, состояние задач и WebSocket, падения consumer/parser tasks (через debug-события в Operator).
4. **Снижение нагрузки от меток времени парсера**: троттлинг `_mark_updated` в базовом классе парсера.
5. **Исправление порядка и блокировок при debug**: отправка в Telegram **не ждёт** тяжёлой сериализации; debug-события **планируются фоном** (`_schedule_debug_log`), чтобы не голодать event loop и не блокировать сигналы.
6. **Перезапуск мёртвых parser-task** и **watchdog по stale** для **agg_trades** / **liquidations** с backoff/jitter.
7. **`mark_running()`** у парсера: после принудительного рестарта сбрасываются метки времени, чтобы watchdog не считал данные вечно «старыми» из-за старого `last_update_ts`.
8. **Connectivity layer**: ожидание интернета при типичных сетевых сбоях (БД, unicex, и т.д.).
9. Правки в **`generate_text`** (корректный вывод CMC rank в заголовке при `daily_signal_count == 1`).
10. Устранение падения **FundingRateParser** из-за несуществующего `repr` (в актуальной ветке).

## 4. Известные и потенциальные проблемы

### 4.1 Эксплуатация и среда

- **Нестабильный или цензурируемый канал (в т.ч. РФ):** задержки, блокировки отдельных хостов, TLS-аномалии. Реализованы ожидание и повтор, список probe-URL можно переопределить; **гарантии доставки данных с бирж нет**, если режется именно трафик до бирж/Telegram.
- **CoinMarketCap:** периодические ошибки SSL/HTTP (`requests`) — рейтинги не обновляются, в UI остаётся `N/A`; на фильтры скринера это обычно не влияет.
- **Telegram:** при лимитах/бане бота сигналы не уйдут; в логах будут ошибки отправки.

### 4.2 Архитектура данных

- **Общие парсеры на пару биржа+рынок:** все скринеры с одной парой делят один **AggTrades** и один набор WebSocket. Высокая нагрузка или отладка нескольких скринеров усиливает конкуренцию за event loop и CPU; часть проблем «пропали сигналы у всех кроме OI» связана с тем, что **OI** обновляется по REST-циклу, а **pump/liquidations** сильнее зависят от **klines** и WebSocket.
- **Обязательность klines:** пока по символу нет свечей из agg_trades, цепочка до многих фильтров не проходит (`no klines data`). Это ожидаемо по текущей логике, но ухудшает UX для скринеров «только OI» при мёртвом agg stream.

### 4.3 Низкий уровень / библиотеки

- В логах Docker возможны сообщения вида **`Task was destroyed but it is pending`** от **unicex/WebSocket._worker** при обрыве или перезапуске — следствие жизненного цикла задач; цель — минимизировать за счёт `stop()` + отмены задач, полностью убрать без правок unicex сложно.
- **Гонки при жёстком рестарте парсера:** кратковременно возможны пустые/частичные данные; consumer должен пережить это за счёт следующих циклов.

### 4.4 Технический долг / ограничения снимка

- В `pyproject.toml` версия всё ещё **0.1.0** — не отражает «v1.2» бэкапа; при релизах лучше синхронизировать.
- В репозитории могут быть **дубликаты/опечатки** в коде парсеров (например, фрагменты в `agg_trades.py`, опечатка `self._logger. info` в liquidations) — при мердже из бэкапа стоит сверить с основной веткой.
- Файл **`.env` в бэкапе**: не коммитить в публичный git; при необходимости хранить только шаблон без секретов.

## 6. Краткая рекомендация по восстановлению из бэкапа
